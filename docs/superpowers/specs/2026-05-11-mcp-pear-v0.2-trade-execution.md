# mcp-pear v0.2 — Trade Execution Design

**Status:** approved 2026-05-11
**Scope:** add write tools (open / close / adjust positions, place / cancel limit / TP / SL / TWAP orders, leverage, risk parameters, agent wallet provisioning) behind a hard `PEAR_TRADE_ENABLED=true` gate
**Audience:** implementation plan author + future contributors

---

## 1. Goal

mcp-pear v0.1.x is read-only — agents can browse markets and read your account, but they can't move funds. v0.2 adds the write tools needed for an agent to actually trade pair positions on Pear Protocol.

The single most important non-functional requirement: **an LLM cannot accidentally execute a trade**. Every write tool short-circuits to a `ConfigError` unless the operator has explicitly opted in via `PEAR_TRADE_ENABLED=true`. Read tools are unaffected — v0.2 is strictly additive.

This is a **minor version bump** (0.1.x → 0.2.0) because the tool surface grows and the env-var contract changes (new mandatory gate for writes).

---

## 2. Verified facts about Pear's trade API

Confirmed by reading `docs.pearprotocol.io/api-integration/api-specification/{positions,orders,agent-wallet}.md` on 2026-05-11.

### Authentication model

**Trades are not client-signed.** Every write endpoint uses the same `Authorization: Bearer <JWT>` header as the read endpoints. Pear holds an *agent wallet* per user (server-side, encrypted) and signs Hyperliquid transactions on the user's behalf after the user approves the agent wallet on Hyperliquid out-of-band.

This means **mcp-pear v0.2 does not need viem for trade signing.** viem stays a dependency only for the v0.1.1 `setup` CLI's EIP-712 verification.

### Endpoints

| Method | Path | Purpose |
|---|---|---|
| `GET`    | `/agentWallet`                              | Get current agent wallet address (may be empty) |
| `POST`   | `/agentWallet`                              | Create a new agent wallet — user then approves on Hyperliquid |
| `POST`   | `/positions`                                | Open a new position (any executionType) |
| `POST`   | `/positions/{positionId}/close`             | Close a position (MARKET or TWAP) |
| `POST`   | `/positions/close-all`                      | Close all open positions |
| `POST`   | `/positions/{positionId}/adjust`            | Reduce / increase position size |
| `POST`   | `/positions/{positionId}/adjust-leverage`   | Change leverage (1–100x) |
| `PUT`    | `/positions/{positionId}/riskParameters`    | Set / update TP / SL on existing position |
| `DELETE` | `/orders/{orderId}/cancel`                  | Cancel pending limit / TP / SL order |
| `POST`   | `/orders/{orderId}/twap/cancel`             | Cancel a TWAP order + all pending chunks |

### `POST /positions` body shape

The single most complex DTO in the API. Fields (per OpenAPI):

```ts
{
  // Required for all:
  executionType: "SYNC" | "MARKET" | "TRIGGER" | "TWAP" | "LADDER" |
                 "TP" | "SL" | "SPOT_BUY" | "SPOT_SELL";
  leverage: number;        // 1–100
  usdValue: number;        // ≥ 1
  longAssets:  Array<{ asset: string; weight: number }>;
  shortAssets: Array<{ asset: string; weight: number }>;
  slippage: number;        // 0.001–0.1

  // Optional / conditional:
  triggerValue?: number;     // TRIGGER
  triggerType?: "ABOVE" | "BELOW"; // TRIGGER
  direction?: "LONG" | "SHORT";   // TRIGGER
  twapDuration?: number;          // TWAP — seconds
  twapIntervalSeconds?: number;   // TWAP
  randomizeExecution?: boolean;   // TWAP
  ladderConfig?: { /* not documented in detail */ };
  stopLoss?: TpSlThreshold | null;
  takeProfit?: TpSlThreshold | null;
  referralCode?: string;
}

type TpSlThreshold = {
  type: "PRICE" | "PERCENTAGE";
  value: number;
  isTrailing?: boolean;
  trailingDeltaValue?: number;
  trailingActivationValue?: number;
}
```

Response: `{ orderId: string, fills: Array<{...}> }`.

### `POST /positions/{positionId}/close` body shape

```ts
{
  executionType: "MARKET" | "TWAP";  // required
  twapDuration?: number;             // required if TWAP
  twapIntervalSeconds?: number;
  randomizeExecution?: boolean;
  referralCode?: string;
}
```

Response: `{ orderId, executionTime, chunksScheduled }`.

### `POST /positions/{positionId}/adjust` body shape

```ts
{
  adjustmentType: "REDUCE" | "INCREASE";
  adjustmentSize: number;            // 1–100 (percent)
  executionType: "MARKET" | "LIMIT";
  limitRatio?: number;               // required if LIMIT
  referralCode?: string;
}
```

### `POST /positions/{positionId}/adjust-leverage` body shape

```ts
{ leverage: number }   // 1–100
```

### `PUT /positions/{positionId}/riskParameters` body shape

```ts
{
  stopLoss?: TpSlThreshold | null;
  takeProfit?: TpSlThreshold | null;
}
```

### Cancel endpoints

`DELETE /orders/{orderId}/cancel` — no body. Response `{ orderId, status, cancelledAt }`.

`POST /orders/{orderId}/twap/cancel` — no body. Response `{ orderId, status, cancelledAt }`.

### Known unknowns

- `ladderConfig` shape isn't documented in detail. We expose it as a passthrough `Record<string, unknown>` and let users supply it raw, with a doc-only hint.
- `adjust-advance` endpoint takes "Array of strings" — undocumented, likely internal. Excluded from v0.2.

## 3. Tool surface

10 new MCP tools. **Every write tool short-circuits with `ConfigError` unless `PEAR_TRADE_ENABLED=true`.**

| Tool | Endpoint | Description |
|---|---|---|
| `get_agent_wallet`        | GET `/agentWallet`                            | **Read.** Get the user's current agent wallet address (may be empty). Recommended first call before any trade. |
| `create_agent_wallet`     | POST `/agentWallet`                           | **Write.** Create a new agent wallet. Returns the address and instructions for user to approve on Hyperliquid. |
| `open_position`           | POST `/positions`                             | **Write.** Open a new pair position. Supports MARKET, LIMIT (TRIGGER), TWAP, LADDER, TP/SL execution types. Required: leverage, usdValue, longAssets, shortAssets, slippage. |
| `close_position`          | POST `/positions/{positionId}/close`          | **Write.** Close a position by ID. MARKET or TWAP execution. |
| `close_all_positions`     | POST `/positions/close-all`                   | **Write.** Close every open position with a single execution type. |
| `adjust_position`         | POST `/positions/{positionId}/adjust`         | **Write.** Reduce or increase a position's size by 1–100%. MARKET or LIMIT. |
| `adjust_leverage`         | POST `/positions/{positionId}/adjust-leverage`| **Write.** Set leverage 1–100x on an existing position. Carries liquidation risk. |
| `set_risk_parameters`     | PUT `/positions/{positionId}/riskParameters`  | **Write.** Set or update TP/SL on an existing position. |
| `cancel_order`            | DELETE `/orders/{orderId}/cancel`             | **Write.** Cancel a pending limit / TP / SL order. |
| `cancel_twap_order`       | POST `/orders/{orderId}/twap/cancel`          | **Write.** Cancel a TWAP order and all of its remaining chunks. |

**Not in v0.2** (deferred to v0.2.x):
- `POST /orders/spot` — different endpoint family; can ship in v0.2.1 once the position-management surface stabilizes.
- `adjust-advance` — undocumented.
- Candle synthesis from Hyperliquid `candleSnapshot` — separate concern, was on the v0.2 roadmap but better-fit for v0.3 with WebSocket streaming.

## 4. The `PEAR_TRADE_ENABLED` safety gate

A hard kill-switch enforced at the tool-execute boundary.

```ts
// src/lib/config.ts (new)
requireTradeEnabled(toolName: string): void {
  if (process.env.PEAR_TRADE_ENABLED !== "true") {
    throw new ConfigError(
      `\`${toolName}\` executes real trades on Pear Protocol / Hyperliquid. ` +
      `It is disabled by default. Set PEAR_TRADE_ENABLED=true in your env to enable.`
    );
  }
}
```

**Rules:**

1. Every write tool calls `getConfig().requireTradeEnabled(toolName)` as its **first action** in `execute()`, before any network call.
2. The gate is checked on every call, not memoized. (Cheap, and lets users toggle the env var live in dev.)
3. Only literal string `"true"` enables. Empty, `"false"`, `"1"`, `"yes"` — all disabled. Strictness here is intentional: ambiguous truthy values are accident vectors.
4. Read tools (v0.1) are not affected.
5. `get_agent_wallet` is treated as a **read tool** (GET) — no gate. `create_agent_wallet` is a write — gated.

**Tool descriptions** prominently warn: every write-tool's description string ends with `WRITE: executes real trades. Requires PEAR_TRADE_ENABLED=true.` so an LLM reading the tool list sees the warning before invoking.

## 6. Data flow (representative — `open_position`)

```
LLM invokes open_position(args)
  │
  ├─ inputSchema.parse(args)        # zod validates leverage 1-100, usdValue ≥1, etc.
  │
  ├─ getConfig().requireTradeEnabled("open_position")
  │     → throws ConfigError if PEAR_TRADE_ENABLED ≠ "true"
  │
  ├─ PearClient.getInstance().createPosition({ ...args })
  │     POST /positions  (Authorization: Bearer <JWT>)
  │     ← { orderId, fills }
  │     reuses authedFetch (JWT ladder, 401 retry, schema validation)
  │
  └─ renderResponse(`Opened position ${shorten(orderId)}…`, body)
```

All write tools follow this exact shape — gate → schema → call → render. Predictable, easy to audit, easy to test.

## 7. Error handling

Inherits v0.1's existing pattern (`formatErrorMessage` in `lib/format.ts`), plus:

| Failure | Tool behavior |
|---|---|
| `PEAR_TRADE_ENABLED` not set | `ConfigError` with explicit opt-in instruction |
| `PEAR_API_KEY` / `PEAR_JWT` missing | `ConfigError` (existing handling) |
| 400 (Pear validation) | Surfaces `HttpError.body.message` verbatim — Pear's error messages are descriptive |
| 401 | Retried by `authedFetch`; if persistent, the existing "Authentication failed" message |
| 403 (e.g. agent wallet not approved) | `formatErrorMessage` surfaces Pear's message verbatim. Tool description warns about the approval prereq. |
| 429 | Retried with exponential backoff via existing `fetchJsonWithRetry` |
| Other 5xx | Retried; if persistent, error surfaces |

**No new error types.** `ConfigError` covers the gate; `HttpError` covers the rest.

## 8. Testing strategy

**Unit tests (TDD per file):**

| File | Coverage |
|---|---|
| `tests/config.test.ts` (extend) | `requireTradeEnabled` enables on `"true"`, throws on absent / `"false"` / `"True"` / `"1"` / `"yes"` |
| `tests/pear-client.test.ts` (NEW) | Each of the 10 new methods: URL, method, body shape, response parsing. Use the existing `vi.stubGlobal("fetch", ...)` pattern. |
| `tests/tools/<each>.test.ts` (NEW per write tool) | Gate behavior (throws when env unset, passes through when set), schema validation, error path returns formatted message instead of throwing |
| `tests/schemas.test.ts` (extend) | TpSlThreshold, PairAssetDto, CreatePositionResponse, ClosePositionResponse, AdjustPositionResponse, CloseAllPositionsResponse parse correctly |

**Live smoke tests (`tests/smoke.test.ts` extension):**

All gated on `PEAR_TRADE_SMOKE=true` (separate from `PEAR_TRADE_ENABLED` — smoke tests must opt-in explicitly even when trade is enabled). When set:

| Test | What it does |
|---|---|
| `get_agent_wallet smoke` | Calls GET /agentWallet — read-only, always safe to run |

**Not live-tested in CI** (would move real funds):
- `open_position`, `close_position`, `adjust_*`, etc. — these must be tested by the maintainer on testnet or with minimal usdValue, not in automated suites.

## 12. Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| LLM accidentally invokes a write tool | high (this is the whole motivation for the gate) | `PEAR_TRADE_ENABLED=true` opt-in; descriptive WRITE warnings in every write tool's description; default-off |
| `PEAR_TRADE_ENABLED=true` set in CI / shared env by accident | medium | env-only, not a CLI flag; documented as "operator opt-in, never check this into version control"; mcp-pear logs `PEAR_TRADE_ENABLED=true; trade execution unlocked` to stderr on server boot when enabled so operators see it |
| Agent wallet not approved on Hyperliquid → first trade fails with cryptic 403 | high | `create_agent_wallet`'s response includes Pear's `message` field verbatim, which contains approval instructions; `get_agent_wallet` callable any time to verify |
| Position-management endpoints' DTO drift | medium | passthrough zod schemas for response bodies (`.passthrough()` on the outer object) so unknown fields don't break parsing |
| `ladderConfig` shape unknown | low | exposed as opaque `Record<string, unknown>` with a doc note; LADDER execution type can be flagged "advanced" in the tool description |
| Server-side signing means user's keys are never on their machine — security model trust | inherent | documented in README; same trust model as `app.pear.garden` itself; not a regression from v0.1 |

## 13. Implementation phasing

Given the surface size and the irreversibility of write tools, v0.2 implementation lands in **two atomic tranches** so each tranche can be reviewed independently:

**Tranche A — foundation (no writes yet):**
1. `requireTradeEnabled` gate + tests
2. Agent wallet schemas + PearClient methods
3. `get_agent_wallet` tool (read-only)
4. `create_agent_wallet` tool (first write — proves the gate works end-to-end)
5. README v0.2 section

**Tranche B — full write surface:**
6. Remaining 8 write tools, each a separate commit (request schema + PearClient method + tool + unit test → one commit)
7. Changeset
8. Bump to 0.2.0
9. Manual smoke against live API (operator runs `create_agent_wallet`, verifies on Hyperliquid; runs `open_position` with small `usdValue`)
10. **STOP before npm publish.** v0.2 is held for explicit operator approval before going public — irreversible if buggy.

Each tranche produces working, shippable software on its own; only the npm publish step is held.
